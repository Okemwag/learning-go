## Using `go run` to Try Out Small Programs

`go run` is the fastest way to try a small Go program without manually creating a binary.

Examples:

```bash
go run ./intermediate/08_go_tooling
go run ./basics/12_functions
```

Use `go run` for:

- experiments
- learning examples
- quick utilities
- verifying a small package behaves as expected

Use `go build` when you need a binary artifact.

## Adding Third-Party Tools with `go install`

For command-line tools, `go install` is the correct modern command.

Example:

```bash
go install golang.org/x/tools/cmd/goimports@latest
```

This installs the tool binary into your Go bin directory.

### Compare with `go get`

`go get` is for managing module dependencies in your current module.

`go install module/path@version` is for installing executables, especially tools, without modifying your current `go.mod`.

Rule of thumb:

- use `go install` for tools
- use `go get` for dependencies

## Improving Import Formatting with `goimports`

`goimports` does two things:

- formats Go code like `gofmt`
- adds/removes imports automatically

That makes it better than plain `gofmt` when you are actively editing imports.

Typical usage:

```bash
goimports -w file.go
```

This is one of the most practical day-to-day Go tools.

## Using Code-Quality Scanners

Go's standard tooling is strong, but external linters catch additional issues.

### `staticcheck`

`staticcheck` is one of the highest-signal Go analyzers.

It catches:

- likely bugs
- suspicious code patterns
- deprecated APIs
- performance and correctness issues

Typical install:

```bash
go install honnef.co/go/tools/cmd/staticcheck@latest
```

### `revive`

`revive` is a configurable linter often used for style and maintainability rules.

Typical install:

```bash
go install github.com/mgechev/revive@latest
```

### `golangci-lint`

`golangci-lint` is a linter runner that combines many linters under one command.

It is common in CI because it:

- centralizes configuration
- runs multiple checks together
- standardizes lint output

## Using `govulncheck` to Scan for Vulnerable Dependencies

`govulncheck` scans your code and dependencies for known vulnerabilities relevant to actual call paths.

Typical install:

```bash
go install golang.org/x/vuln/cmd/govulncheck@latest
```

Typical run:

```bash
govulncheck ./...
```

This is more useful than a raw dependency list because it considers how code is actually used.

## Using `go generate`

`go generate` runs commands described by `//go:generate` comments.

In this lesson, the source includes:

```go
//go:generate sh -c "printf 'Generated by go generate\n' > generated.txt"
```

Then run:

```bash
go generate ./intermediate/08_go_tooling
```

Important:

- `go generate` is not part of `go build`
- it must be run explicitly

## Working with `go generate` and Makefiles

In larger projects, a Makefile often coordinates generation and builds:

```make
generate:
	go generate ./...

build:
	go build ./...
```

This keeps routine workflows consistent in CI and for contributors.

## Reading the Build Info Inside a Go Binary

`runtime/debug.ReadBuildInfo()` can expose build metadata embedded in a Go binary.

This is useful for:

- support diagnostics
- version reporting
- understanding module provenance

The example program in this lesson prints the main module path and version when available.

## Building Go Binaries for Other Platforms

Go cross-compilation is straightforward.

Examples:

```bash
GOOS=linux GOARCH=amd64 go build ./intermediate/08_go_tooling
GOOS=windows GOARCH=amd64 go build ./intermediate/08_go_tooling
GOOS=darwin GOARCH=arm64 go build ./intermediate/08_go_tooling
```

This is one of Go's strongest operational advantages.

## Using Build Tags

Build tags let you include files only for certain conditions.

Typical uses:

- platform-specific code
- optional integrations
- debug-only behavior

They are declared at the top of files with:

```go
//go:build linux
```

Use build tags carefully. They are powerful, but too many can fragment the codebase.

## Testing Versions of Go

A practical way to test multiple Go versions is to run CI across a version matrix.

This matters because:

- language features depend on version
- tool behavior may differ
- dependency compatibility may shift

For local work, version managers such as `asdf`, `gvm`, or toolchain downloads can help.

## Using `go help` to Learn More about Go Tooling

The built-in help system is extremely useful.

Examples:

```bash
go help
go help build
go help modules
go help generate
go help test
```

This is often the fastest way to get precise command behavior directly from the toolchain.

## Deep Note

Go tooling is strongest when you lean into the defaults: use `go run` and `go build` for core workflows, add a small set of high-signal external tools, automate generation and linting explicitly, and rely on the built-in help system instead of memorizing everything. The more your project uses standard Go tooling conventions, the easier it is for other Go developers to work in it.
